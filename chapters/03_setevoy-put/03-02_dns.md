## 3.2. Разрешение имени хоста: DNS

Прежде чем установить TCP-соединение, клиенту нужно превратить имя `srv-1c.company.local` в IP-адрес. Это задача **DNS** (Domain Name System) — той самой распределённой телефонной книги интернета, которая работает с 1983 года и обслуживает сегодня весь мир.

1С здесь не делает ничего своего. Клиент вызывает стандартную системную функцию Windows `getaddrinfo` (или её аналог в более старом API — `gethostbyname`), и дальше всё происходит ровно так же, как для любой другой программы, которой нужно куда-то подключиться по сети. Ниже — то, что делает Windows под капотом, без участия 1С:

**Шаг 1. Кеш DNS-резолвера в самой Windows.** Windows держит в памяти небольшой кеш недавних DNS-ответов. Если за последние несколько минут какая-то программа (или сама Windows) уже разрешала это имя, ответ берётся из кеша мгновенно, без сетевого запроса. Посмотреть содержимое кеша можно командой `ipconfig /displaydns`, очистить — `ipconfig /flushdns`. Если ответа в кеше нет, идём дальше.

**Шаг 2. Файл `hosts`.** Windows смотрит в файл `C:\Windows\System32\drivers\etc\hosts`. Это локальный «телефонный справочник», в котором можно вручную прописать соответствие имени и IP. Если для нашего имени там есть запись — она используется, и DNS-запрос вообще не отправляется. Это древний механизм (наследие ARPANET до появления DNS), но он по-прежнему работает и часто применяется для отладки, для блокировки рекламы, для тестовых обходов корпоративных DNS, и так далее.

**Шаг 3. Запрос к DNS-серверу.** Если ни в кеше, ни в `hosts` ответа нет, Windows формирует UDP-запрос (или TCP, для больших ответов) к одному из настроенных DNS-серверов. Какие DNS-серверы использовать — Windows знает из настроек сетевого адаптера: они либо получены автоматически по DHCP (типичный случай в офисной сети), либо прописаны вручную, либо назначены через VPN. В корпоративной среде первым DNS-сервером обычно выступает **контроллер домена**, который держит зону `company.local` и знает, что `srv-1c.company.local` — это, например, `192.168.10.50`.

**Шаг 4. Рекурсивное разрешение.** Если локальный DNS-сервер не знает имени напрямую (нечастый случай в корпоративной среде, потому что внутренние имена обычно лежат на нём же), он рекурсивно запрашивает другие DNS-серверы, пока не найдёт ответ. Это та самая «классическая» процедура DNS-резолвинга, которую Алекс Гейнор подробно описывает в `what-happens-when` для случая публичного интернета.

**Шаг 5. Возврат IP-адреса клиенту.** Когда ответ получен, Windows кладёт его в свой кеш с TTL (временем жизни, заданным DNS-сервером — обычно несколько минут или часов), и возвращает IP-адрес клиенту 1С через `getaddrinfo`. С этого момента клиент знает, что `srv-1c.company.local` — это, скажем, `192.168.10.50`, и можно открывать TCP-соединение.

> **Параллель.** Это та же цепочка, что для любого другого сетевого приложения: браузера, почтового клиента, SSH. Для диагностики это удобно: стандартные сетевые утилиты (`nslookup`, `ping`, `tracert`, `Wireshark`) работают с трафиком 1С точно так же, как с любым другим — никаких специальных инструментов не нужно.

> **Если что-то пошло не так.**
>
> - **«Не удалось разрешить имя сервера».** DNS-резолвинг не смог найти `Srvr`. Возможные причины: опечатка в имени; компьютер не в нужной сети (например, не подключён к VPN, и потому не видит внутренний DNS); упал DNS-сервер; проблемы с маршрутизацией. Самый быстрый способ диагностики — открыть командную строку и выполнить `nslookup srv-1c.company.local`. Если `nslookup` тоже не находит — проблема не в 1С, а в DNS, и решать её нужно средствами сети.
> - **DNS отвечает, но IP «не тот».** Имя резолвится в неправильный адрес. Чаще всего — старая запись в файле `hosts`, прописанная вручную ещё до переезда сервера. Реже — устаревшая запись в кеше или на DNS-сервере. Лечится правкой `hosts` и/или `ipconfig /flushdns`.
> - **Клиент работает «по IP», но не работает «по имени»** (или наоборот). Если вы можете подключиться, прописав в `Srvr` IP-адрес, но не можете по имени — это явная DNS-проблема. Обратная ситуация (по имени работает, по IP нет) встречается реже и обычно связана с тем, что сертификат TLS или Kerberos жёстко привязаны к имени.
