## 9.4. Где в 1С своё, а где — мировые стандарты

В течение всей статьи мы по ходу замечали, что одни вещи в 1С — это собственные форматы и протоколы, а другие — стандартная индустриальная инфраструктура. Соберём это наблюдение в одной таблице — не как манифест, а как полезный ориентир для понимания, где платформа использует общеизвестные технологии (и значит, можно искать ответы в общем интернете), а где специфические знания применимы только к 1С.

| Уровень | Что используется |
|---|---|
| Запуск программы (часть I) | Стандартный механизм Windows: `ShellExecute`, `CreateProcess`, PE-формат, DLL search order |
| Лаунчер (часть II) | Собственный формат `.v8i` (концептуально близкий к INI), собственный `1cestart.cfg`, собственный механизм автообновления через `DistributiveLocation` |
| Сетевой транспорт (часть III) | Стандартные TCP, DNS, HTTP, TLS — без всякой 1С-специфики. Системные API: `getaddrinfo`, Winsock, Schannel |
| Прикладной протокол сервера 1С (часть IV) | **Собственный** бинарный протокол поверх TCP. Не документирован публично. Между клиентом, `ragent`, `rmngr` и `rphost` ходит специфика 1С |
| СУБД (часть IV) | Стандартные СУБД: MS SQL, PostgreSQL/Postgres Pro, Oracle, Db2. PostgreSQL — в специальной сборке с патчами (нормальная индустриальная практика, не уникальная для 1С). Поверх этого — **собственный язык запросов 1С**, который транслируется в SQL средствами платформы |
| Файловая база (часть IV) | **Собственный формат** `1Cv8.1CD` — embedded-БД с собственной страничной структурой и блокировками |
| Веб-сервер (часть IV) | Стандартные Apache и IIS со стандартными механизмами модулей. Плюс собственный встроенный веб-сервер для разработки |
| Веб-протоколы поверх HTTP (часть IV) | Стандарты: HTTP, REST, SOAP, OData. Внутри них — частично собственный бинарный протокол 1С (для тонкого клиента через `ws://`) |
| Аутентификация ОС (часть V) | Стандартные SSPI, Kerberos, NTLM. Никакой собственной криптографии для опознания пользователя |
| OpenID (часть V) | Стандартный протокол OpenID |
| Хранение пользователей и хешей паролей (часть V) | **Собственный механизм** в служебных таблицах ИБ. Алгоритм хеширования паролей — стандартный криптографический (соль + хеш-функция), но конкретные детали — внутренняя кухня |
| HASP-лицензии (часть V) | Стандартный продукт Sentinel HASP от Thales (бывший SafeNet/Aladdin), используемый сотнями коммерческих приложений. Не специфика 1С |
| Программные лицензии (часть V) | **Собственный формат** `.lic` файлов с собственной системой привязки к железу |
| Метаданные конфигурации (часть VI) | **Собственный формат** хранения, собственная структура объектов, собственные правила компиляции |
| Расширения (часть VI) | **Собственный механизм**, не имеющий прямого аналога в других платформах |
| RBAC + RLS (часть VI) | Концепции стандартные (RBAC — везде, RLS — в современных СУБД), но в 1С реализованы **на уровне платформы поверх СУБД**, со своим синтаксисом шаблонов |
| Встроенный язык 1С и его компилятор (часть VII) | **Собственный** язык и **собственная** виртуальная машина с собственным байт-кодом. По модели похожа на JVM или CPython, но реализация полностью своя |
| Управляемые формы (часть VIII) | **Собственный механизм** декларативного описания форм, собственный движок рендеринга |
| Толстый клиент Windows (часть VIII) | Стандартные WinAPI-контролы — нативное Windows-приложение |
| Тонкий клиент Linux/macOS (часть VIII) | Qt — стандартная кросс-платформенная UI-библиотека |
| Веб-клиент (часть VIII) | Стандартные HTML, CSS, JavaScript, DOM. Само JavaScript-приложение веб-клиента — собственная разработка 1С |

### Общая картина

Если посмотреть на эту таблицу, видно довольно ясный паттерн: **1С использует индустриальные стандарты везде, где они есть и подходят, и имеет собственные форматы там, где специфика учётной системы требует чего-то своего**.

Транспорт, аутентификация, СУБД, рендеринг UI — везде, где есть готовые мировые решения, 1С их использует. Это и логично (нет смысла переизобретать TCP), и практически выгодно (можно пользоваться готовой инфраструктурой, инструментами диагностики, навыками индустрии).

С другой стороны, **прикладная модель** — то, ради чего платформа существует — это собственная разработка от и до. Объекты конфигурации, язык запросов 1С, управляемые формы, расширения, права с RLS, встроенный язык — всё это заточено под предметную область учётных систем, и аналогов в индустрии им либо нет, либо они работают по другим принципам. Здесь 1С предлагает свою архитектуру, и эта архитектура имеет свои сильные и слабые стороны, о которых разработчики спорят десятилетиями.

Это нормальное распределение для специализированной платформы: **инфраструктура — стандартная, прикладная модель — собственная**. То же самое можно сказать про большинство крупных корпоративных решений (SAP, Oracle E-Business Suite, Microsoft Dynamics): транспорт у всех общий, БД у всех стандартные, а вот «как описывается заказ покупателя» — у каждого своё, и это и есть основная ценность продукта.
