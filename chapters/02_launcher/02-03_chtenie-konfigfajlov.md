## 2.3. Чтение конфигурационных файлов

Теперь лаунчер начинает читать настройки. У него есть **два типа конфигурационных файлов**, и каждый существует в нескольких экземплярах в разных каталогах. Понять эту систему — значит понять, как 1С раздаёт настройки в больших организациях.

**Файл `1cestart.cfg`** — основной файл настроек лаунчера. В нём могут быть указаны:

```
InstalledLocation=C:\Program Files\1cv8\8.3.22.1923\
InstalledLocation=C:\Program Files\1cv8\8.3.23.2137\
DefaultVersion=8.3.22.1923
CommonInfoBases=\\fileserver\1c\common.v8i
CommonCfgLocation=\\fileserver\1c\1CEScmn.cfg
DistributiveLocation=\\fileserver\1c\distr\
UseHwLicenses=1
ProxyUseSystemSettings=1
```

Каждая строка — отдельный параметр. Перечислим самые важные:

- **`InstalledLocation`** — пути к установленным версиям платформы. Может быть несколько строк. Лаунчер сам обнаруживает версии в `Program Files`, но через этот параметр можно явно указать дополнительные места.
- **`DefaultVersion`** — какую версию использовать по умолчанию, если у конкретной базы версия не задана.
- **`CommonInfoBases`** — путь к **общему списку баз**. Может быть локальным файлом, сетевым путём (`\\server\share\common.v8i`) или HTTP-адресом. Это главный механизм централизованной раздачи списка баз сотрудникам.
- **`CommonCfgLocation`** — путь к ещё одному конфигурационному файлу (`1CEScmn.cfg`), который содержит общие настройки для всей организации. Через него администраторы раздают единые настройки прокси, политику обновлений и так далее.
- **`DistributiveLocation`** — где лежат дистрибутивы платформы для автоматической установки. Очень важный параметр: благодаря ему тонкий клиент может **сам себя обновить**, если сервер требует более новой версии. Подробнее в пункте 2.7.
- **`UseHwLicenses`** — использовать ли аппаратные ключи защиты (HASP). По умолчанию да.
- **`ProxyUseSystemSettings`** — брать ли настройки прокси из системы Windows.

**Файл `1CEScmn.cfg`** — общий конфигурационный файл для централизованных настроек. Формат тот же, состав параметров пересекается с `1cestart.cfg`. Идея в том, что `1CEScmn.cfg` обычно лежит в сетевом каталоге и поддерживается администратором, а `1cestart.cfg` — локальный и может быть у каждого пользователя свой.

**Где лаунчер ищет эти файлы.** Лаунчер последовательно проверяет несколько мест, причём порядок имеет значение — настройки **сливаются** с приоритетом в пользу более «локальных» источников:

| Приоритет | Файл | Где лежит | Кто обычно правит |
|---|---|---|---|
| 1 (низший) | `1CEScmn.cfg` | Каталог, указанный в `CommonCfgLocation` (обычно сетевой) | Администратор организации |
| 2 | `1cestart.cfg` | `%ALLUSERSPROFILE%\1C\1CEStart\` | Администратор машины |
| 3 (высший) | `1cestart.cfg` | `%APPDATA%\1C\1CEStart\` | Сам пользователь |

Чем выше приоритет источника, тем сильнее он переопределяет более низкие. Это даёт удобную трёхуровневую модель настроек: организация → машина → пользователь. Администратор организации может в `1CEScmn.cfg` задать единый список общих баз для всех — а конкретный пользователь может в своём `1cestart.cfg` добавить ещё несколько личных, и они появятся в списке вместе с общими.

> **Параллель.** Это та же идея, что в Git с тремя уровнями конфигурации — `--system`, `--global`, `--local`. Или в `.editorconfig` с наследованием от корня файловой системы вниз. Или в Maven `settings.xml` (глобальный и пользовательский). Везде один принцип: общие настройки задаются централизованно, локальные могут их переопределить, и явное переопределение всегда побеждает.
