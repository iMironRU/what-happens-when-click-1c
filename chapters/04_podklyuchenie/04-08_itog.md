## 4.8. Практический разворот: как от строки подключения дойти до физической базы

Зачем это нужно знать. Представьте: вы — администратор организации, и у вас есть запись в `.v8i`:

```
Connect=Srvr="srv-1c.company.local";Ref="accounting_prod";
```

Вам нужно:

- Сделать резервную копию этой базы (внепланово, не дожидаясь штатного бэкапа).
- Перенести базу на другой сервер для тестирования.
- Понять, сколько места она занимает и где именно.
- Восстановить её после сбоя.

Куда идти? Что копировать? Где смотреть? Эта часть проводит вас по всей цепочке от строки подключения до физических данных.

### Файловая база — простой случай

Для файловой базы всё прозрачно: путь в строке подключения = каталог на диске.

```
Connect=File="\\fileserver\1c\accounting";
```

→ База лежит в `\\fileserver\1c\accounting\`. Всё содержимое каталога — это и есть база. Главный файл — `1Cv8.1CD`. Для резервной копии достаточно скопировать каталог целиком.

**Важное правило:** файловую базу нельзя копировать «горячей» — то есть пока с ней работают пользователи. Нужно дождаться, пока все сеансы закроются, и только потом копировать. Иначе можно скопировать файл в полу-обновлённом состоянии (в момент, когда платформа писала какие-то страницы), и копия будет повреждённой.

Для серверной базы всё намного интереснее.

### Серверная база — пройти всю цепочку

Дано: `Srvr="srv-1c.company.local";Ref="accounting_prod"`. Что делать?

**Шаг 1. Найти физический хост, на котором живёт сервер 1С.** В простой топологии это сам `srv-1c.company.local`. В распределённом кластере с несколькими серверами — нужно идти в консоль кластера и смотреть, на каком из физических серверов запущен `rphost`, обслуживающий именно эту базу. Обычно это любой из серверов кластера, но конкретные `rphost` могут быть назначены определённым хостам через механизм требований назначения функциональности.

**Шаг 2. Найти каталог `srvinfo` агента сервера на этом хосте.** Каталог `srvinfo` — это «база данных» агента сервера 1С, в которой хранится вся информация о кластерах, базах, регистрациях, сессиях. Путь к нему задаётся при установке сервера 1С — параметром при старте службы `ragent`. По умолчанию это что-то вроде:

```
C:\Program Files\1cv8\srvinfo\
```

или на Linux:

```
/home/usr1cv8/.1cv8/1C/1cv8/
```

Точный путь у конкретной установки может быть другим — администратор задал его при установке. Узнать его можно либо из параметров службы Windows (`ragent.exe -d <путь>`), либо просто поискав каталог `srvinfo` на дисках сервера.

**Шаг 3. Внутри `srvinfo` — структура реестра кластера.** Она выглядит примерно так:

```
srvinfo/
└── reg_1541/                                  ← реестр кластера на порту 1541
    ├── 1CV8Reg.lst                            ← список зарегистрированных баз
    ├── 1CV8Clst.lst                           ← список кластеров
    └── <UUID-базы>/                           ← каталог конкретной базы
        ├── 1Cv8Log/                           ← журнал регистрации этой базы
        ├── snccntx<UUID>/                     ← контекст сеансов
        └── ... другие служебные файлы
```

**Файл `1CV8Reg.lst`** — это «каталог» всех баз, зарегистрированных в кластере. Внутри в собственном текстовом формате 1С — список баз с их параметрами, включая UUID и параметры подключения к СУБД. Вот что для нас важно: для каждой базы там лежит её **строка подключения к СУБД** — имя сервера БД, имя БД на этом сервере, имя пользователя и пароль (последний — в зашифрованном виде).

**Шаг 4. Извлечь из `1CV8Reg.lst` параметры подключения к СУБД.** Открыв `1CV8Reg.lst` (это текстовый файл, но формат специфический), можно найти секцию для нашей базы `accounting_prod` и увидеть что-то вроде: тип СУБД — `PostgreSQL`, сервер БД — `db-postgres.company.local`, имя БД — `accounting_prod_db`, пользователь — `usr1c`. Пароль зашифрован, но для бэкапа он нам и не нужен — мы уже знаем достаточно.

**Альтернативный путь, который проще.** Вместо того чтобы лезть в файлы `srvinfo` руками, открыть эти же параметры можно через **консоль администрирования сервера 1С**: подключиться к кластеру, найти базу в списке, открыть её свойства — там в человекочитаемом виде показаны все параметры подключения к СУБД. Это удобнее и не требует понимания формата `1CV8Reg.lst`. Прямое чтение `srvinfo` нужно только в редких случаях, когда консоль недоступна или когда нужно автоматизировать получение этих параметров скриптом.

**Шаг 5. Идти в СУБД.** Теперь мы знаем, где физически лежат данные: в БД `accounting_prod_db` на сервере `db-postgres.company.local` под СУБД PostgreSQL. Дальше работаем уже инструментами этой СУБД.

### Три способа сделать резервную копию серверной базы

**Способ 1: средствами СУБД.** Это самый правильный, быстрый и универсальный способ. Используются штатные инструменты конкретной СУБД:

- **PostgreSQL:** `pg_dump` для логического дампа (текстовые SQL-команды), `pg_basebackup` для физической копии всего кластера PostgreSQL, плюс настроенное архивирование WAL для point-in-time recovery.
- **MS SQL Server:** `BACKUP DATABASE` с файлом `.bak`, плюс настроенные планы обслуживания SQL Server Management Studio.
- **Oracle:** RMAN.
- **Db2:** утилиты Db2 backup.

Преимущества: быстро, можно делать «на горячую» (без выгона пользователей), интегрируется со штатными системами резервного копирования организации, поддерживает инкрементальные и дифференциальные бэкапы. Недостатки: бэкап привязан к конкретной СУБД и часто к её версии. PostgreSQL-бэкап не восстановишь в MS SQL.

**Способ 2: выгрузка в `.dt` через Конфигуратор.** Это «логический» бэкап средствами самой платформы 1С. Открыть базу в Конфигураторе, Администрирование → Выгрузить информационную базу, указать путь к файлу. Получаем файл `.dt`, в котором лежат и конфигурация, и все данные в собственном формате 1С (специальный архивный формат, переносимый между СУБД и версиями платформы).

Преимущества: переносимо между любыми поддерживаемыми СУБД (можно выгрузить из MS SQL и загрузить в PostgreSQL), переносимо между версиями платформы (с некоторыми ограничениями для очень старых версий), не требует доступа к СУБД, делается через привычный для одинэсника интерфейс.

Недостатки и они серьёзные:
- **Требует монопольного доступа.** Все пользователи должны выйти из базы — выгрузка не работает с активными сеансами. Для рабочей базы это означает остановку работы.
- **Очень медленно на больших базах.** На базе в 100 ГБ выгрузка может занимать часы. На 500 ГБ — день и больше. На терабайтных базах формально работает, на практике неприменим.
- **Требует много места во временных файлах** во время выгрузки.

`.dt` хорош для маленьких баз, для переносов с СУБД на СУБД, для отправки базы партнёру или в техподдержку. Для регулярного бэкапа production — категорически нет.

**Способ 3: утилита миграции от фирмы «1С».** Отдельный инструмент, который умеет переносить информационную базу напрямую с одной СУБД на другую (или в ту же самую) **на уровне кластера, без промежуточного `.dt`-файла**. Решает главную боль `.dt` — необходимость огромного промежуточного файла и долгое время выполнения. Используется в первую очередь для миграций (типичный сценарий: «MS SQL → PostgreSQL» при переходе организации с американской СУБД на российскую), но фактически даёт ещё один способ сделать «логическую» копию.

### Какой способ когда выбирать

| Задача | Способ |
|---|---|
| Регулярный production-бэкап серверной базы | Средствами СУБД |
| Перенос базы между разными СУБД | `.dt` или утилита миграции |
| Перенос маленькой базы между организациями | `.dt` |
| Бэкап файловой базы | Копирование каталога целиком |
| Восстановление на конкретный момент времени (PITR) | Только средствами СУБД с архивацией WAL/log |
| Миграция большой production-базы между СУБД | Утилита миграции |

**Главная мысль для запоминания.** Резервное копирование серверной базы 1С — это **резервное копирование БД на стороне СУБД**, а не «копирование файлов 1С». Каталог `srvinfo` хранит только метаданные кластера 1С (учёт сеансов, журналы, параметры подключения к СУБД) — собственно данных вашей базы там нет. Если вы скопируете `srvinfo` целиком, у вас будет описание «на каком сервере жил кластер» — но не сами данные. Данные лежат в СУБД, и копировать их нужно средствами этой СУБД.

> **Параллель.** Это та же модель, что у любого приложения, работающего с внешней БД: бэкап делается на двух уровнях. **Логический** — средствами приложения, медленно, но переносимо (`.dt` в 1С, `pg_dump` для PostgreSQL в SQL-режиме). **Физический** — средствами СУБД, быстро, но привязано к движку (`pg_basebackup`, MS SQL `BACKUP DATABASE`). 1С даёт оба варианта плюс отдельный инструмент миграции для случаев, когда даже логического бэкапа уже не хватает.
