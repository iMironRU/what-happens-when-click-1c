## 4.5. Веб-публикация: три способа

Веб-публикация — это способ выставить информационную базу через HTTP, чтобы к ней могли подключаться тонкий клиент через `ws://`, веб-клиент через браузер, и внешние системы через HTTP/SOAP/OData-сервисы (всё это мы уже обсуждали в части III).

Технически это работает так: на веб-сервере (IIS, Apache или встроенный) включается специальный модуль платформы 1С, который умеет принимать HTTP-запросы и транслировать их в кластер 1С. Этот модуль конфигурируется через **файл `default.vrd`**, который генерируется утилитой `webinst.exe` при публикации.

### Что делает `webinst.exe`

`webinst.exe` — консольная утилита из комплекта платформы. Запускается с параметрами «опубликуй базу X на веб-сервере Y по адресу Z». Она:

1. Создаёт каталог публикации (например, `C:\inetpub\wwwroot\accounting\` для IIS) и кладёт в него файл `default.vrd`.
2. Прописывает в конфигурации веб-сервера (для IIS — через метабазу IIS, для Apache — в `httpd.conf` или `apache.conf`) виртуальный каталог или приложение, указывающее на этот каталог.
3. Регистрирует в веб-сервере, что обращения к этому пути должны обрабатываться модулем 1С (`wsisapi.dll` для IIS или соответствующим модулем Apache).
4. Прописывает в `default.vrd` параметры подключения к информационной базе: какой кластер 1С, какая база, какие сервисы (HTTP, SOAP, OData) включены.

После этого по адресу `https://server/accounting/` начинает работать веб-публикация.

**Файл `default.vrd`.** Это XML-файл с настройками публикации. Внутри — те же самые `Srvr` и `Ref`, что и в строке подключения у клиента, плюс флаги «включить веб-клиент», «включить HTTP-сервисы», «включить OData», «включить SOAP», и так далее. Можно править руками, но проще через `webinst.exe` или через диалог публикации в Конфигураторе.

### Три варианта веб-сервера

**1. Apache HTTP Server.** Кросс-платформенный веб-сервер, работает и на Windows, и на Linux. Это главное практическое преимущество Apache для 1С: серверная часть платформы давно и хорошо работает на Linux, и в современных установках (особенно российских, где импортозамещение давно стало нормой) Linux-серверы — скорее правило, чем исключение. Apache на Linux-сервере с установленным сервером 1С — самая распространённая «взрослая» конфигурация веб-публикации сегодня. Под Windows Apache тоже работает, но там у него есть конкурент в лице IIS, и выбор обычно зависит от привычек администратора.

Платформа 1С предоставляет для Apache свой модуль, который подключается к веб-серверу через стандартный механизм Apache-модулей. Настройка делается через `webinst.exe` (или его аналог на Linux), который правит конфигурационные файлы Apache и кладёт нужные файлы публикации в указанное место.

**2. IIS (Microsoft Internet Information Services).** Штатный веб-сервер Windows, идёт в комплекте с ОС (нужно только включить компонент). Для 1С работает через ISAPI-модуль `wsisapi.dll`. Применим только на Windows — у IIS нет версии для Linux. Встречается на установках, где Windows-сервер уже есть и где Apache никто настраивать не хочет: «у нас Windows-инфраструктура, в IIS администратор разбирается, IIS и используем». Это рабочий и отлаженный вариант, особенно для веб-публикации внутри организации.

**3. Встроенный веб-сервер платформы.** Это особый вариант, который часто упускают из внимания. Платформа 1С содержит **собственный** простенький веб-сервер, который можно использовать без Apache и без IIS. Запускается командой публикации с указанием порта, после чего платформа сама слушает HTTP-запросы на указанном порту и обрабатывает их.

Встроенный веб-сервер не предназначен для production-нагрузки (он простой и не имеет тех оптимизаций, что у промышленных веб-серверов), но он невероятно удобен для:

- **Разработки и отладки.** Разработчику нужно потестировать HTTP-сервис или мобильное приложение, обращающееся к 1С. Поднять Apache или IIS на ноутбуке ради этого — оверкилл. Запустил встроенный веб-сервер на 8080 — и всё работает.
- **Малых установок.** Если веб-публикация нужна для пары пользователей и не критична, встроенного веб-сервера хватает.
- **Файловых баз для разработки.** Это ключевое — встроенный веб-сервер позволяет легко опубликовать **файловую** базу через HTTP без поднятия отдельного веб-сервера.

### Публикация файловой базы через веб

Это часто удивляет: «как файловую базу опубликовать через HTTP, если у неё нет сервера 1С?» Ответ — опубликовать может **сам клиентский процесс или встроенный веб-сервер**, потому что для веб-публикации нужны только бинарники платформы (которые есть), а не работающий `ragent`.

Технически это выглядит так: на машине с файловой базой запускается веб-сервер (любой из трёх — IIS, Apache, встроенный), `webinst.exe` публикует файловую базу с указанием пути к каталогу с `1Cv8.1CD`. После этого по HTTP к этой публикации можно подключаться тонким клиентом или браузером.

С точки зрения файловой базы веб-публикация — это просто ещё один клиент, открывающий её в разделённом режиме. Параллельно с веб-публикацией к этому же файлу могут подключаться обычные клиенты — напрямую, по тому же пути в файловой системе. Все они координируются через стандартный механизм блокировок файловой базы (`1Cv8.1CL`), как и любые другие процессы, работающие с файловой базой одновременно. Веб-публикация не получает никакого «эксклюзива» на файл — она работает по тем же правилам, что и прямое подключение.

Зачем это вообще нужно? Несколько практических сценариев:

- **Мобильные приложения, работающие с локальной базой по HTTP.** Удобный способ дать мобильному клиенту доступ к небольшой файловой базе без поднятия серверной части.
- **Простая удалённая работа.** Один пользователь работает с файловой базой через тонкий клиент или браузер по `ws://`. Не нужно покупать лицензию на сервер 1С.
- **Тестирование HTTP-сервисов** при разработке: разработчик пишет HTTP-сервис в файловой базе, поднимает встроенный веб-сервер и тестирует.

> **Если что-то пошло не так с публикацией.**
>
> - **404 на корне публикации.** Веб-сервер запущен, но публикация не видна. Чаще всего — `webinst.exe` отработал с ошибкой или модуль 1С не зарегистрирован в веб-сервере. Для IIS — проверить наличие виртуального каталога в IIS Manager. Для Apache — проверить, что модуль 1С загружен в `httpd.conf`.
> - **500 Internal Server Error.** Веб-сервер вызвал модуль 1С, но модуль упал. Чаще всего — `default.vrd` указывает на несуществующий кластер или базу; либо у учётки, под которой работает веб-сервер (IUSR в IIS, www-data в Apache), нет прав на каталог публикации или на каталог с файловой базой.
> - **Публикация работает, но подключиться тонким клиентом не получается.** Браузер на корень публикации открывает страницу — значит сетевая часть и веб-сервер в порядке. Тонкий клиент через `ws://` — нет. Часто причина в том, что `webinst.exe` выключил подключение тонкого клиента в `default.vrd` (есть отдельный флаг) — проверить и включить.
