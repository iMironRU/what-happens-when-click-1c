## 4.2. Клиент-серверная база: цепочка `ragent` → `rmngr` → `rphost`

Теперь — то, ради чего вообще существует серверный вариант 1С. В этом режиме платформа разделяется на несколько отдельных процессов с чётко разными ролями, и клиенту нужно пройти через цепочку из них, прежде чем добраться до данных.

Вспомним, где мы остановились в части III: у клиента есть установленное TCP-соединение с портом 1540 на сервере, указанном в `Srvr`. На том конце соединения слушает первый из трёх ключевых процессов сервера 1С — **агент сервера**.

### Три процесса серверной части

**`ragent` (агент сервера 1С).** Это служба Windows (или systemd-юнит на Linux), которая запускается при старте операционной системы и работает постоянно. Её задача — управлять остальными процессами сервера 1С на этом хосте. Сама по себе `ragent` не обслуживает базы и не работает с данными — он только запускает, перезапускает и останавливает другие процессы по командам менеджера кластера. По умолчанию слушает порт **1540**. На один хост приходится ровно один `ragent` — это «корень» серверной части 1С на машине.

**`rmngr` (менеджер кластера).** Это «мозг» кластера 1С. Он знает, какие информационные базы существуют в кластере, какие сейчас активны, какие рабочие процессы (`rphost`) их обслуживают, какие сеансы подключены, какие блокировки удерживаются. Менеджер ведёт учёт сеансов, балансирует нагрузку, принимает решения о запуске новых рабочих процессов или остановке старых. По умолчанию слушает порт **1541**. Запускается агентом при старте кластера.

**`rphost` (рабочий процесс).** Это собственно «исполнитель» — процесс, в котором работает прикладной код конфигурации, формируются ответы на запросы клиентов, происходит обмен с СУБД. Один кластер обычно держит несколько `rphost` параллельно: они разделяют между собой нагрузку, и если один упадёт, остальные продолжат работать. Каждый `rphost` слушает свой динамически назначенный порт из диапазона **1560–1591** по умолчанию (диапазон настраивается). Запускаются менеджером по требованию: когда нужно обслуживать новые сеансы или когда текущие `rphost` перегружены.

> **Параллель.** Эта трёхуровневая структура с разделением «супервизор / координатор / рабочие» — классический паттерн серверного ПО. Очень похоже на устройство Apache HTTP Server в prefork-режиме (главный процесс + рабочие воркеры), на nginx (master + workers), на PostgreSQL (postmaster + backend-процессы), на PHP-FPM (master + pool workers). Идея везде одна: один процесс-родитель управляет жизненным циклом рабочих процессов, рабочие изолированы друг от друга и могут падать независимо. У 1С добавлен один уровень — отдельный менеджер кластера, потому что в кластере может быть несколько физических серверов с несколькими `ragent`, и нужен координатор, знающий обо всех них.

### Что происходит при подключении клиента

Теперь шаг за шагом. У нас есть TCP-соединение клиента с `ragent` на порту 1540.

1. **Клиент представляется агенту.** По соединению клиент отправляет первый запрос на собственном бинарном протоколе 1С: «я хочу подключиться к информационной базе с именем `accounting_prod`». Здесь же передаются версия клиента, тип клиента (тонкий/толстый), и другие технические параметры.

2. **Агент перенаправляет к менеджеру.** `ragent` сам не знает про базы — он просто диспетчер. Он либо передаёт запрос менеджеру кластера (`rmngr` на порту 1541), либо сообщает клиенту адрес менеджера и говорит «иди туда». Реализация этого шага скрыта внутри протокола, и снаружи это выглядит как одна непрерывная сессия с сервером.

3. **Менеджер ищет базу в кластере.** `rmngr` смотрит в свои внутренние структуры: «есть ли в этом кластере база с именем `accounting_prod`?» Если нет — клиент получает ошибку «информационная база не обнаружена в кластере». Если есть — менеджер должен выбрать рабочий процесс, который будет её обслуживать.

4. **Менеджер выбирает или запускает `rphost`.** Если для этой базы уже есть запущенные рабочие процессы и они не перегружены, менеджер выбирает один из них (по балансировке, см. ниже). Если активного `rphost` для базы нет или все перегружены — менеджер даёт команду агенту запустить новый `rphost`. Агент стартует новый процесс, тот рапортует менеджеру о готовности, менеджер заносит его в свой реестр.

5. **Менеджер возвращает клиенту адрес `rphost`.** Это конкретный TCP-адрес: имя или IP сервера и динамический порт из диапазона 1560–1591. С этого момента менеджер из общения выпадает — его задача выполнена.

6. **Клиент устанавливает прямое соединение с `rphost`.** Это **второе TCP-соединение** того же клиента, теперь уже на конкретный порт рабочего процесса. По нему пойдёт всё дальнейшее общение с базой: загрузка метаданных, запросы данных, вызовы процедур, передача форм. Соединение долгое, на всю длительность сеанса.

7. **`rphost` подключается к СУБД** (если ещё не подключён) и начинает обслуживать сеанс. Подробнее про это — в пункте 4.4.

С этого момента общение клиента и `rphost` идёт напрямую, без участия `ragent` и `rmngr`. Клиент шлёт запросы рабочему процессу, тот их выполняет (и при необходимости ходит в СУБД), результаты возвращаются обратно. Менеджер кластера в фоне продолжает следить за состоянием рабочих процессов: считает нагрузку, замеряет память, может перезапустить `rphost` если он съел слишком много памяти, может перенести сеансы между процессами.

> **Если что-то пошло не так.**
>
> - **«Сервер 1С:Предприятия недоступен».** Самая частая ошибка, и она покрывает целое семейство причин: TCP до порта 1540 не доходит, либо `ragent` не отвечает на запросы, либо отвечает, но дальше что-то ломается. Диагностика — как в части III: сначала проверяем сеть до 1540, потом смотрим, запущена ли служба сервера на хосте.
> - **«Информационная база не обнаружена в кластере».** TCP-соединение прошло, агент и менеджер ответили, но имя базы в строке подключения (`Ref="accounting_prod"`) менеджеру неизвестно. Опечатка в имени, или базу удалили из кластера, или клиент подключается не к тому кластеру (если на одном хосте несколько кластеров на разных портах).
> - **«Ошибка получения адреса рабочего процесса»** или **тайм-аут при попытке подключиться к `rphost`**. Менеджер вернул адрес рабочего процесса, но клиент не может к нему подключиться. Чаще всего — диапазон портов 1560–1591 заблокирован фаерволом между клиентом и сервером. Это классическая ошибка настройки сети: открыли только 1540, забыли про диапазон рабочих процессов.
