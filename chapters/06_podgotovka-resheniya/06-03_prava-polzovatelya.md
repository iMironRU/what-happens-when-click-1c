### 6.3. Расчёт прав пользователя

Метаданные загружены, расширения применены. Теперь у `rphost` есть полное и окончательное описание конфигурации, с которым он будет работать. Следующий вопрос — **что именно нашему Иванову можно делать в этой конфигурации**.

#### Роли и права

В 1С система прав построена вокруг понятия **роли**. Роль — это именованный набор разрешений на действия с объектами конфигурации:

- На уровне **объекта** (справочника, документа, регистра): можно ли читать, добавлять, изменять, удалять, проводить (для документов), смотреть в подсистеме.
- На уровне **реквизита** объекта: можно ли видеть и менять конкретное поле (например, «Зарплата сотрудника» в справочнике «Сотрудники» доступна только роли «Бухгалтер по зарплате»).
- На уровне **табличной части**: аналогично реквизитам.
- На уровне **операций**: можно ли использовать определённые действия — открывать Конфигуратор, запускать тонкий клиент, выполнять обмен данными, и так далее.
- На уровне **подсистем**: видеть ли определённый раздел в командном интерфейсе.

Роли описываются в метаданных конфигурации (то есть приходят вместе с самой конфигурацией; разработчик типовой создаёт их при разработке решения, и они применяются у всех клиентов). Типичная типовая конфигурация содержит десятки и сотни ролей: «Полные права», «Только просмотр», «Менеджер по продажам», «Кассир», «Бухгалтер материалов», и так далее.

**Пользователю ИБ назначается одна или несколько ролей**. В свойствах пользователя в Конфигураторе администратор отмечает галочками, какие роли ему доступны. Если ролей несколько, права **объединяются** по принципу «разрешено, если разрешено хотя бы в одной роли». То есть если у пользователя есть роль «Менеджер» (которая разрешает читать справочник Контрагентов) и роль «Кассир» (которая запрещает читать его), итоговое право — **разрешено**.

> **Параллель.** Это абсолютно классическая модель Role-Based Access Control (RBAC), которую вы видели во множестве систем — от Active Directory и Linux-групп до AWS IAM и Kubernetes RBAC. Везде один принцип: набор именованных ролей, каждая роль содержит разрешения, пользователь получает роли, итоговые права — объединение прав всех ролей. 1С здесь идёт по проторенной дорожке.

#### RLS — права на уровне строк

До этого момента всё было довольно стандартно. Но у 1С есть один механизм, выходящий за рамки обычного RBAC, — **RLS (Row-Level Security)**, ограничения доступа на уровне отдельных записей.

Идея в том, что иногда «можно или нельзя» зависит не от объекта в целом, а от **содержимого конкретной записи**. Типичные примеры:

- Менеджер по продажам видит только тех контрагентов, которые закреплены за ним.
- Бухгалтер видит только документы своей организации (если в базе ведётся учёт по нескольким юрлицам).
- Кладовщик видит только остатки своего склада.

Все три случая нельзя выразить через «разрешено читать справочник или нет» — потому что разрешено не «весь справочник» или «весь склад», а «только некоторые записи». Для этого в роли описывается **шаблон ограничения** — фрагмент кода на языке запросов 1С, который определяет, **какие именно записи** видит пользователь с этой ролью.

Шаблон выглядит примерно как фрагмент SQL-WHERE, только на синтаксисе языка запросов 1С. Например, для роли «Менеджер по продажам» ограничение на справочник Контрагентов может быть таким:

```
ГДЕ Контрагенты.ОтветственныйМенеджер = &ТекущийПользователь
```

Здесь `&ТекущийПользователь` — параметр запроса, который берётся из параметра сеанса (вспоминаем часть V, пункт 5.3 — теперь понятно, зачем там это всё было).

#### Как RLS превращается в SQL

Когда пользователь делает запрос «дай мне всех контрагентов», прикладной код пишет:

```
ВЫБРАТЬ * ИЗ Справочник.Контрагенты
```

Платформа берёт этот запрос и **автоматически добавляет** к нему все RLS-ограничения, действующие для текущего пользователя. Если у пользователя есть роль «Менеджер по продажам» с ограничением выше, итоговый запрос превращается во что-то вроде:

```
ВЫБРАТЬ * ИЗ Справочник.Контрагенты
ГДЕ Контрагенты.ОтветственныйМенеджер = &ТекущийПользователь
```

И уже этот расширенный запрос транслируется в SQL и отправляется в СУБД. Пользователь не видит, что к запросу что-то добавилось — он просто получает «всех контрагентов», которых ему положено видеть. Если у него несколько ролей с разными ограничениями, они объединяются по тому же принципу «разрешено, если разрешено хотя бы в одной»: пользователь видит запись, если её показывает **хотя бы одна** из его ролей.

Этот процесс происходит **на каждом запросе**, для каждого объекта, к которому идёт обращение. Шаблоны RLS компилируются заранее (в момент расчёта прав пользователя), а потом подставляются в запросы при их выполнении.

> **Параллель.** RLS как концепция есть в современных СУБД (PostgreSQL поддерживает Row Security Policies начиная с 9.5, Oracle — Virtual Private Database, SQL Server — Row-Level Security). Но в 1С эта функциональность реализована **на уровне платформы**, поверх СУБД, и не зависит от её возможностей: даже если бы 1С работала с СУБД, не поддерживающей RLS на своём уровне, всё равно бы работала. Платформа сама добавляет ограничения в запросы перед их трансляцией.

#### Кеширование рассчитанных прав

Расчёт прав — относительно дорогая операция: нужно собрать все роли пользователя, объединить их разрешения, скомпилировать шаблоны RLS с подстановкой параметров сеанса, и подготовить всё это к подстановке в будущие запросы. Делать это на каждый запрос было бы расточительно.

Поэтому права рассчитываются **один раз на сеанс** и кешируются в памяти `rphost` на всё время жизни сеанса. Когда сеанс закрывается, кеш освобождается.

Из этого следует одна важная практическая деталь: **изменения ролей или назначений пользователю не подхватываются на лету**. Если администратор добавил пользователю новую роль в Конфигураторе или в режиме Предприятия, текущий сеанс этого пользователя продолжит работать со старыми правами — кеш ведь рассчитан раньше. Чтобы новые права вступили в силу, **пользователь должен перезайти**. Это поведение часто удивляет администраторов: «я же дал ему права!» — а пользователь отвечает «у меня всё ещё нет». Лечение универсальное — выйти и зайти заново.

Аналогично, изменение значений параметров сеанса, на которые опираются шаблоны RLS, тоже требует учитывать кеш. Если параметр сеанса меняется в процессе работы (через `УстановитьПараметрыСеанса`), кеш RLS-условий, использующих этот параметр, должен инвалидироваться — иначе пользователь продолжит видеть данные «по старому». Платформа это умеет, но прикладному коду нужно знать о таких нюансах.

#### Если что-то пошло не так с правами

> - **«У пользователя недостаточно прав на выполнение операции».** Самое частое сообщение системы прав. Пользователь пытается сделать что-то, на что у его ролей нет разрешения. Решается либо назначением подходящей роли, либо корректировкой существующей.
> - **«Запись не найдена»** или **в списке не видно записей, которые точно есть в базе**. Классическая ситуация с RLS: записи существуют, но текущему пользователю не разрешено их видеть. Часто это законное поведение, иногда — ошибка в шаблоне RLS или в данных (например, у контрагента не заполнен «ответственный менеджер», и ни один менеджер его не видит).
> - **Изменения прав не применились.** Администратор поменял роли, но пользователь продолжает работать со старыми правами. Лечится перезаходом пользователя в базу.
> - **Сеанс падает с ошибкой в шаблоне RLS.** Если шаблон ссылается на параметр сеанса, который не установлен, или на реквизит, которого нет в текущей версии конфигурации, — компиляция RLS падает, и сеанс не стартует. Лечится правкой роли в Конфигураторе.
